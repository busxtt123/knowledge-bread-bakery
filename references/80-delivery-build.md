# 80 出柜交付

## 0. 核心结论

交付不是把过程文件丢给顾客，而是把面包摆上柜台。默认最终交付适合手机阅读的 PDF；顾客明确要求 Markdown 时，可以同时交付整理后的 `delivery/final.md`。内容形态仍然只有单篇长文或多讲课程，PDF 和 Markdown 只是交付格式。

阅读体验优先。技术验证只说明文件生成成功，不说明它已经好读。排版也关系顾客能不能读懂：它决定顾客是否愿意读下去、能不能快速找到层级、会不会在密集信息里疲劳。

默认 PDF 的审美方向是“安静的精心准备”：像一份认真交到顾客手里的正式成品，而不是从 Markdown 直接打印出来的文件。

可以借鉴四类经典排版思路：

- **书籍排版**：稳定的版心、克制的标题、舒服的行距，让读者忘记排版本身。
- **瑞士网格**：层级清楚，留白有纪律，不用装饰抢注意力。
- **Tufte 式内容优先**：视觉设计服务信息，不为好看制造噪声。
- **移动阅读器**：窄版心、短行长、段落呼吸，让手机上连续阅读不累。

落到默认 PDF，先守五条：

1. **正文先稳**：字号、行距、版心先让连续阅读舒服，其他设计都要退后。
2. **层级少而清**：标题负责导航，不用重边框、大色块或过多装饰证明“有设计”。
3. **留白有纪律**：页边距、段间距、列表间距稳定，读者才不用重新适应每一页。
4. **特殊块降噪**：引用、表格、代码块只提示信息类型，不抢正文重量。
5. **封面要精心**：封面和目录要像交付物，不像自动导出的第一页；但不加入与内容无关的品牌装饰。

## 1. 最终文稿

从 `workspace/manuscript.md` 整理出 `delivery/final.md`。整理时只做必要统一：标题层级、衔接、目录、重复、未回收伏笔、明显语病。不要把每一讲磨成同一种声音。

标题层级先过一遍：一级标题只给整本产品标题，而且要像顾客会看到的书名或课程名，不要写成 `# 课程正文`、`# 最终稿`、`# final` 这类后台名称。封面副标题必须写，且只用一种写法：放在一级标题下一行的引用行里，例如：

```markdown
# 为什么
> 因果科学入门课
```

副标题是封面承诺，不进入正文目录。它要用一句话说明这份作品会带读者进入什么理解，不要写成后台说明，也不要把 `# 标题：副标题` 当作副标题格式。

发刊词、导论、前言和各讲正文都是正文目录项，统一用二级标题；导论内部的小节和讲内小标题用三级标题或正文加粗。不要让 `delivery/final.md` 以 `# 发刊词`、`# 导论` 或 `# 前言` 开头。

前置内容标题必须保持二级。`### 前言`、`### 发刊词` 或 `### 导论` 不是“小节”，会让 PDF 目录和分页错层；标准脚本会在生成前报出具体行号。遇到这类错误，修 `delivery/final.md`，不要改脚本绕过去。

出炉前做一次书籍结构体检。体检只看成品骨架，不审内容优劣：

- 只有一个一级标题，且一级标题是整本产品名称。
- 一级标题下一行有 `> 副标题`，且副标题像给读者看的封面承诺。
- PDF 目录只收二级标题，所以二级标题必须像书的骨架：发刊词 / 导论 / 第 N 讲 / 第 N 章 / 结语 / 结课地图等。
- 已经有“第 N 讲 / 第 N 章”结构时，讲内小标题不要用二级标题；否则它会混进目录，让成品不像一本书。
- 生成前看一眼脚本打印的 `PDF 目录预览`。如果这份列表不像读者会翻阅的目录，先回 `delivery/final.md` 修层级。

检查：

- 标题层级清楚。
- 段落适合手机阅读。
- 没有生产过程中的残留标记。
- 关键判断都有出处。
- 每章 / 每讲有可读的进入和收束。

## 2. 一键生成 PDF

只保留一条标准路线。`delivery/final.md` 第一行就是整本产品标题，脚本从这里取封面和 PDF 元数据标题：

```bash
python scripts/build-mobile-pdf.py delivery/final.md delivery/final.pdf
```

脚本做的事：Markdown → HTML → WeasyPrint PDF。负责：

- 接近手机竖屏比例的页面尺寸。
- 暖白纸色、柔和正文色和克制强调色。
- 手机阅读友好的版心、字号、行距和段落间距。
- 封面和目录不显示页码，正文页才显示页码。
- 目录和二级标题锚点。
- CJK 字体栈。
- 保留正文里中英文交界处的作者空格，不擅自把 `PDF 单一出口` 挤成 `PDF单一出口`。
- 列表、引用、表格、代码块基础排版。
- 可复制正文。
- 生成前做书籍结构体检，发现缺少封面副标题、标题层级和目录边界问题时给出具体行号。
- 打印 `PDF 目录预览`，让出炉前能看见哪些标题会进入目录。

依赖缺失时，脚本会在 skill 目录下创建 `.skill-runtime/pdf-venv`，安装 `markdown` / `weasyprint` 并重新运行。不要再临时创建第二套 PDF 脚本；如果标准脚本不满足需求，优先修 `scripts/build-mobile-pdf.py`，让系统继续保持一个出口。

标准脚本只依赖 Python 包 `markdown` 和 `weasyprint`，不依赖 `markdown_it`。如果报错出现 `ModuleNotFoundError: No module named 'markdown_it'`，说明运行的不是当前标准脚本，或脚本被旧版本覆盖；先恢复 / 修正 `scripts/build-mobile-pdf.py`，不要为这次交付另写临时脚本。

## 3. PDF 验收

技术验收：

- `delivery/final.pdf` 存在且非空。
- `pdfinfo delivery/final.pdf` 可读取页数和尺寸。
- `pdftotext delivery/final.pdf -` 可抽取正文。
- 抽取的文本没有 `file://`、本地路径、日期页眉页脚等残留。
- PDF 标题是整本产品标题，不是发刊词、导论或前言。
- 封面有副标题；`delivery/final.md` 的一级标题下一行是 `> 副标题`。
- 目录里前置内容和正文章节层级正确：发刊词 / 导论 / 前言不被抽成封面，也不把导论内部小节提升成并列大章。
- 标准脚本的格式预检通过；若报出前置内容标题层级错误，先改 `delivery/final.md`。

阅读验收：

- 目录可跳转。
- 标题层级清楚。
- 字号、行距、页边距舒服，不像挤压后的打印稿。
- 封面安静、有留白，目录像目录，不像普通正文。
- 页面整体安静，不用大面积色块、重边框或过多高亮抢注意力。
- 段落不挤，列表换行对齐。
- 长表格没有横向挤压到不可读。
- 引用、代码块、表格是辅助信息，不喧宾夺主。
- 正文可以复制。
- 手机视角抽查阅读，不只看桌面预览；至少看封面、目录、一个普通正文页、一个列表页、一个表格或引用页。

`pdftotext` 只用于验证文本可抽取、中文无乱码、无 file URL / 本地路径残留。不要用它判断首行缩进、列表对齐、间距等视觉问题；视觉问题用 PDF 阅读器或截图看渲染结果。

## 4. 排版返工信号

- 看起来像 Word / Markdown 直接导出的文档。
- 正文一屏里黑压压一片，没有呼吸。
- 标题装饰太重，抢走正文注意力。
- 每个页面都像套了同一块色板，缺少细腻层级。
- 引用块、表格或代码块比正文更显眼。
- 长表格在手机上不可读；先回到 `delivery/final.md`，把表格拆成分组列表或卡片式表达，这不是 CSS 能解决的。
- PDF 好看但不像本产品：颜色、装饰或字体气质盖过了内容。

## 5. 非默认交付

过程材料、结构抽取、课程规划和创作日记可以作为生产现场保留，但不默认当作最终交付。顾客明确要“给创作继续用的材料”时，仍要整理成可读、可接手、边界清楚的文件，而不是把过程文件原样打包。
