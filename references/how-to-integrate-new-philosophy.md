# 如何把新思想融进面包房

## 0. 这页是什么

这页是 skill 维护规则，不是内容生产 SOP。只有在顾客要求把新的创作理念、设计哲学、外部方法论或系统治理原则融进 skill 时，才读它。

它存在的理由很简单：升级 skill 不是把新词塞进旧流程，而是让新思想改变某个工位的判断、动作、验收或返工路线。没有改变生产行为的新增文字，只会变成信息垃圾。

使用它时，先判断新思想到底解决哪类失败，也判断它希望系统长出什么能力、作品抵达什么状态，再决定改哪个文件；改完后做一致性复查，确认没有平行规则、重复解释和旧说法残留。

## 1. 文档治理

盖尔系统从一个能跑的简单系统长出来，不从一堆看起来完整的规则长出来。维护 skill 时先问：这个新增判断会让哪个工位少犯哪一种错，还是只让文档看起来更用力？后者不要写进系统。

每个文档都按“是什么、为什么、怎么用”层层展开：

- **是什么**：它承担哪一种职责，是主流程、工位 SOP、方法页、标准页、异常页，还是维护页。
- **为什么**：它为什么需要独立存在，解决哪个别的文件解决不了的问题。
- **怎么做**：读者拿到它以后该怎样使用，读哪些文件，写到哪里，如何判断完成或返工。

不同职责有不同重心。标准页可以少讲背景，多讲通过 / 不通过；异常页可以少讲常规流程，多讲第一反应和接管判断；方法页可以少讲派工，多讲判断结构。但一个文件如果只靠“这里也有相关内容”存在，就不该保留原状。

同一层级尽量 MECE：先给结论，再展开理由、边界和动作；同类判断放在同类文件，避免一个原则散落成几处相似提醒。MECE 是为了减少误会，不是为了把手艺做成填表。

引用链接要减少上下文负担，而不是制造跳转迷宫。一个文件通过链接仍然说不清自己的作用，或者和其他文件承担同一件事，就按顺序处理：

1. **融合**：把重复判断并回真正负责的文件。
2. **重新定位**：缩小职责，改成概念页、方法页或异常页。
3. **删除**：如果没有独立职责，也没有保留价值，删除并同步清理引用。

## 2. 分层放置

新思想进入系统时，先确定它改变哪一层：

- **主手册 `SKILL.md`**：只放主流程、闸门、派工边界和返工路线。不要在这里复写工位 SOP。
- **工位页 `references/agents/*.md`**：放“这个工位怎么干”。读取文件的动作要融进步骤，并说明读它形成什么判断。
- **方法页 `references/*.md`**：放跨工位可复用的判断方法、产品原则、形态判断、方向治理或异常处理。不能变成第二套流程。
- **标准页 `references/standards/*.md`**：放通过标准、走偏气味和返工方向。标准页不教完整操作。
- **脚本和交付页**：只放稳定、可复用、能减少人工不确定性的技术路径。

如果一个新思想同时影响多层，必须让每一层只承担自己的那部分。不要把同一段解释复制到四个地方。

## 3. 改写步骤

### 3.1 先找原因与目标

动手前先写出一句维护目的：因为系统出现了哪一种偏差，所以要让哪个工位更容易做出哪一种判断，最后让作品更接近什么状态。写不出这句话，就先不要改文件。

先问：

- 它到底防什么偏差？
- 它希望什么状态更多出现？
- 它会改变哪个工位的选择？
- 没有它，产物会出现什么症状？
- 这个症状来自标准不清、职责错位、方向不足、表达失真，还是现场异常？

只有能回答这些问题，才开始改文件。

同时判断这件事属于哪种改变。优先把新思想写成愿景、判断和动作，让执行者知道该朝哪里用力；只有真正会造成生产事故的内容，才写成硬边界。

- **底线**：不这样做会出事故，放进流程、标准或故障页。
- **标准**：不这样做会降低平均质量，放进验收和返工。
- **方向**：这样做会拉高作品上限，放进方法页、创作者工位或品尝标准。
- **维护规则**：只影响 skill 自身怎么改，放在本页，不进入生产流程。

### 3.2 写成愿景驱动

愿景驱动不是把标准放松，也不是把问题写得漂亮一点。它是先告诉执行者这一步为什么值得做、要让什么能力出现、什么样的作品状态算更好，再用标准和边界托住底线。

改写时先写正向目的，再写必要边界。不要让文档满屏都是“不要”“避免”“不是”，却没有说明应该怎样把事情做好。限制如果只是为了防错，就问它背后的作品目标是什么；能转成方向时，优先转成方向。

例如，“不能并行创作”背后的目标不是反对效率，而是让同一位创作者承接前文的语气、判断和余味，让课程像一条连续长出来的路。这样写，执行者知道要保护什么，也知道什么时候可以变通、什么时候不能变通。

愿景要落到动作。只写“追求优秀”没有用；要写清楚哪个工位看什么、形成什么判断、写到哪里，主理人怎样品尝，没做到时回哪一站。

### 3.3 再读同类旧内容

先直接读相关文件的上下文，理解这条规则在现场里承担什么，再用 `rg` 找词和功能补漏：

```bash
rg -n "旧概念词|相关动作|相关文件名" SKILL.md references scripts
```

不要只看字面，也不要用搜索结果替代阅读。旧词可能不同，但如果承担同一件事，就要融合或改写。

### 3.4 决定改写范围

- **只影响表达**：改对应工位页和标准页。
- **影响生产行为**：至少改主手册、工位页、标准页和返工路线。
- **影响系统结构**：同步检查方法页、异常页、项目现场、交付和维护规则。

### 3.5 整段重写

优先整段重写，不做贴补丁式追加。写完每段都检查：

- 这段说明了什么。
- 为什么放在这里。
- 下一步怎么做。
- 做错了回哪里改。

如果回答不上来，删掉或并到真正负责的文件。

### 3.6 写成自然中文

自然中文不是口语化，也不是堆文采。它是用中文本来成立的语法、节奏和语义关系，把判断说顺、说准、说到位。维护 skill 时，语言既要让执行者愿意读，也要让执行者知道下一步怎样做。

一句话先保住主干，再处理修饰。通常先说“谁 / 什么在什么条件下做什么”，再说理由、边界或后果；不要让三个抽象名词互相修饰，却看不见真正的动作。能用动词说清的，不用名词堆叠代替。

中文重承接，也重语势。需要表达因果时，用“因为……所以……”“这会让……”“否则……”；需要表达条件时，用“如果……就……”“只有……才……”；需要表达步骤时，用“先……再……最后……”。关系说清楚，执行者才知道判断从哪里来、下一步往哪里去。

词语要按语境承担任务。一个词如果只是看起来有气势，却没有告诉执行者发生了什么，就换掉。判断“压到”“误伤”“位置被放低”这类说法时，不只看它是否形象，而要看它是否传递了预期效果：是让读者感到被指责，还是让材料的力量变弱，还是让目录层级失真。说不清，就改成更平实的中文。

标题也要自然。标题不是英文概念的硬翻，也不是漂亮标签；它应该让读者一眼知道这一节管什么。能说“出炉前结构体检”，就不要写成“交付完整性防御机制”；能说“写成愿景驱动”，就不要写成“目标导向表达模块”。

模糊可以保留，但要保留得有用。好的模糊给执行者判断空间，比如“让材料出味”“让读者能进入”；坏的模糊遮住主语、动作和后果，比如“优化认知链路”“提升结构体验”。前者帮助智能体结合现场判断，后者只让文本显得正确。

最后读一遍，问三件事：这句话希望读者怎么理解，句子结构是否完整，它是否真的把想表达的判断传过去了。读起来别扭，不一定是错；但如果别扭来自中文里没有这种说法，就重写。

## 4. 复查

改完后做人工复查：

```bash
rg -n "新概念|旧概念|关键动作|相关文件名" SKILL.md references VERSION.md CHANGELOG.md
```

重点看：

- 主流程、工位、标准和返工路线是否一致。
- 是否出现同一规则在多个文件重复展开。
- 是否有旧文件名、旧概念或相反规则残留。
- 是否把私密创作现场、主理人工作记录、验收标准混在一起。
- 是否新增了无法解释独立用途的文档。
- 是否只剩禁令和检查，却没有写出作品目标、工位愿景和正向动作。
- 中文是否顺着语境成立，标题、句子和词语是否真的达成表达目的。

## 5. 典型失败

- **只加一句原则**：主手册写了，工位不知道怎么做，标准不会验收。
- **复制解释**：四个文件都写同一段话，后续改一处必然漏三处。
- **引用逃避**：文件只丢链接，不说自己是什么、为什么存在、怎么用。
- **过度简化**：文字短了，但 subagent、逐讲、品尝、写入边界等关键动作丢了。
- **过度程序化**：动作都在，但没有判断，创作者只会填表。
- **禁令堆叠**：每条都像对，但执行者只知道不要犯错，不知道怎样把作品做得更好。
- **中文失真**：词语看似高级，中文语境里却不成立；句子没有主干，读者无法判断动作、原因和后果。

正确的改法，是让新思想进入已有职责分层：该托底的做标准，该防遗漏的做框架，该拉上限的做方向，该保护现场的做异常处理。
