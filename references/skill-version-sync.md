# Skill 版本发布参考

## 0. 这页解决什么

这页只处理版本发布，不处理日常修订。普通工作区修改先保持可审阅的 diff；用户确认版本号、准备发布、复制到正式目录或同步到外部副本时，再更新版本信息。

它存在的理由是让“正在运行的版本”和“记录里的版本”指向同一次事实。否则后续排查时会分不清：当前 skill 是哪一版，文档记录是哪一版，发布副本又是哪一版。

如果还有项目备份、发布副本或 Obsidian 副本，同步属于发布动作；本页只规定 skill 内部先自洽。

## 1. 什么时候同步

满足任一条件，再同步版本：

- 用户已经确认新版本号。
- 本次改动准备发布或复制到正式 skill 目录。
- 要把工作区副本同步到备份、发布副本或 Obsidian 副本。
- 需要用版本号定位一次可回溯的生产行为变化。

不满足这些条件时，不为了普通文案修订强行改版本号。先保留 diff，让修改内容可审阅。

## 2. 必须同步的五处

- `SKILL.md` frontmatter 的 `version:`
- `VERSION.md`
- `references/VERSION.md`
- `CHANGELOG.md`
- `references/CHANGELOG.md`

五处必须一致。不能只改主入口，也不能只改根目录版本。

## 3. 版本记录写法

`VERSION.md` 和 `references/VERSION.md` 只写版本号，例如：

```text
2.0.0
```

`CHANGELOG.md` 和 `references/CHANGELOG.md` 最新版本写在最上方：

```markdown
# 版本记录

当前版本：2.0.0

## 2.0.0 — 2026-05-16
- 说明这次改变了哪些判断、动作和返工路线。
```

版本记录不要只写”改了措辞”。要写清楚这次改变会怎样影响实际生产。

## 4. 发布副本

如果本 skill 还有另一个备份或发布目录，先在目标侧保留备份，再同步。不要在没有备份的情况下做删除式镜像。

同步后人工检查：

```bash
rg -n "version:|当前版本|^2\\.0\\.0$" SKILL.md VERSION.md references/VERSION.md CHANGELOG.md references/CHANGELOG.md
```

除非用户明确要求，否则不运行一致性脚本；人工核对足够时就停。

## 5. 常见失误

- 改了 `SKILL.md`，忘记改两个 `VERSION.md`。
- skill 根目录 `CHANGELOG.md` 和 `references/CHANGELOG.md` 不一致。
- 版本记录只写理念，没有写影响到哪个工位、动作或返工路线。
- 用户还没确认版本号，就为了普通修订自行升版本。
- 历史措辞残留了已经废弃的流程名，导致 `rg` 看起来像还有旧规则。
- 同步发布副本时把本地运行环境、缓存或临时文件一起带出去。
