# 00 开门清点标准

## 0. 判断目标

开门清点只判断原料是否真的进得了厨房。它不做顾客访谈，不做结构抽取，也不决定最终产品形态。合格状态是：材料找得到、已经转成 Markdown 可读副本、读得到、缺口清楚、来源清单明确，后面的验料同事能从 Markdown 副本开始工作。

材料转换 SOP 写在 `references/agents/00-kitchen-starter.md`。项目现场只记录本单采用了哪些来源、当前是否可读；长期可复用的转换经验直接进入开门清点 SOP 或脚本说明，不在生产流程里另写备忘录。

## 1. 通过标准

### 1.1 项目现场已经在正确时机建立

- 需求配方已通过，正式进入生产后才建立项目现场。
- 项目现场位置清楚：顾客指定文件夹、材料所在目录旁，或当前文件夹下新建的本单文件夹。
- 最小结构已存在：`sources/`、`workspace/`、`delivery/`。
- `workspace/need-recipe.md` 已包含顾客原话、材料入口、信息源、当前缺口和吸收成功标准。

### 1.2 材料入口清楚

- 每份材料的文件、笔记、PDF、EPUB、Word、Markdown、TXT 或资料包位置清楚。
- 已记录材料类型、来源名称、原始路径、文件大小和当前读取状态。
- 已记录 Markdown 可读副本路径；不能转换的材料已写明原因和下一步。
- 没有在开门清点阶段提前记录章节、页码、标题层级、block 或 Markdown 起止行；这些留给验料 / 结构抽取站。

### 1.3 材料真的可读取

- 已亲自打开或抽样读取主要材料，确认 agent 能读到正文，而不只是看到文件名、封面或目录。
- 已判断材料状态：完整 / 缺页 / 扫描件 / 权限不足 / 需要 OCR / 需要导入 / 需要转换。
- PDF、EPUB、Word、Markdown、TXT、PPT、Excel、图片页和已整理成文件的文本材料，已经按 `references/agents/00-kitchen-starter.md` 的材料转换 SOP 处理；OCR 只作为扫描 PDF / 图片页 fallback。
- 网页、微信公众号、YouTube、在线文章和其他 URL 不进入转换流程；顾客只给链接时，已请顾客提供本地文档或电子书副本。
- 不可读材料没有被送入验料。

### 1.4 缺口和下一步清楚

- 缺文件、缺权限、格式不可读、扫描质量差、材料损坏等缺口已写明。
- 若材料经过转换，已重新抽样读取转换结果。
- `sources/source-manifest.md` 已记录每份来源、材料种类、文件大小、Markdown 副本路径、转换状态、可读状态和边界。

## 2. 走偏气味

- 材料在哪里都不清楚就进入阅读或验料。
- 只知道文件存在，没有亲自打开或抽样读取。
- 正文实际不可读，却继续进入结构抽取。
- 可转换来源没有落成 Markdown 副本。
- `sources/source-manifest.md` 缺少材料清单、转换状态或 Markdown 副本路径。
- 把“目标食客 / 产品类型”定死，而没有说明只是初判。
- 正式生产没有项目现场。

## 3. 返工方向

- 缺材料或权限：回到顾客确认。
- 材料不可读：回到开门清点，按 `references/agents/00-kitchen-starter.md` 的材料转换 SOP 处理。
- 来源清单不清：补 `sources/source-manifest.md`；需求配方只保留顾客为什么来、材料入口和吸收成功标准。
- 现场未建：先支项目现场，再重新清点。

只有确认材料能进入 agent 的阅读和结构抽取，才进入验料。
