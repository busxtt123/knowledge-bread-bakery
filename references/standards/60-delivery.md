# 60 交付标准

## 0. 判断目标

交付标准只判断一件事：顾客拿到成品后，能不能直接读、愿意读、读完能用。

文件生成成功只是底线。真正的交付，是内容、边界和阅读体验一起站住。

## 1. 通过标准

- 成品形态匹配顾客真实用途。
- 默认交付适合手机阅读的 PDF；顾客明确要求 Markdown 时，可以同时交付整理后的 `delivery/final.md`。
- 内容形态仍然是单篇长文或多讲课程，PDF 和 Markdown 只是交付格式。
- PDF 由 `scripts/build-mobile-pdf.py` 从 `delivery/final.md` 生成，不临时分叉第二套脚本。
- `delivery/final.md` 只有一个一级标题，且一级标题是整本产品标题，不是 `课程正文`、`最终稿`、`final` 等后台名称。
- `delivery/final.md` 的一级标题下一行必须是 `> 副标题`；副标题是给读者看的封面承诺，不用冒号标题或正文段落代替。
- 不把过程文件当默认交付物。
- 关键判断可追溯到材料，风险边界干净。
- PDF 有目录跳转、清楚标题层级、舒服字号 / 行距 / 页边距、可复制正文。
- `delivery/final.md` 不以 `# 发刊词`、`# 导论` 或 `# 前言` 开头；这些前置内容在正文里使用二级标题，导论内部小节使用三级标题。
- `delivery/final.md` 不把发刊词、导论或前言写成三级标题；标准脚本的标题层级预检通过。
- PDF 目录只收二级标题；已有“第 N 讲 / 第 N 章”结构时，讲内小标题不用二级标题。
- 出炉前看过标准脚本打印的 `PDF 目录预览`，目录列表像一本书或一门课的骨架。
- PDF 元数据和封面标题是整本产品标题，不是前置内容标题。
- PDF 排版服务认知交付：简洁、安静、好看，但不抢正文注意力。
- 手机阅读时段落有呼吸，标题能导航，封面和目录有交付感，引用 / 表格 / 代码块是辅助而不是视觉主角。
- 阅读体验通过手机视角抽查，不只是文件生成成功。
- 至少保留一个挑战既有观念的“不舒服”论点，或说明材料为什么不支持；没有做成低配科普。
- 维护读者认知体验底线：成品没有用指责、羞耻或亏欠启动阅读，也没有把读者写成必须先承认错误的人。
- 没有让后台面包房语言替代前台真实表达。

## 2. 走偏气味

- 默认交一堆过程文件。
- 成品不是顾客能直接阅读或使用的东西。
- 把交付格式或外部包装当成新的内容形态。
- 来源和推断混杂。
- PDF 或文档格式影响手机阅读。
- 一级标题是 `课程正文`、`最终稿`、`final` 等后台名称，读者打开后看不见真正的书名。
- 缺少封面副标题，或用 `# 标题：副标题` 代替 `> 副标题`。
- 发刊词、导论或前言被误当成整本 PDF 标题。
- 发刊词、导论或前言被降成三级标题，导致目录和分页层级错误。
- 在多讲课程里，讲内小标题被写成二级标题，混进 PDF 目录。
- PDF 看起来像直接导出的 Markdown / Word，缺少精心交付感。
- 排版装饰过重，颜色、边框、高亮抢走正文注意力。
- 标题、引用、表格、代码块的视觉重量超过正文。
- 长表格或密集列表在手机上不可读。
- PDF 生成链路分叉，导致后续无法维护或复现。
- 易理解但没劲，只剩顺滑常识。
- 入口或发刊词用指责、羞耻或亏欠启动阅读，却没有让张力来自材料、旧模型、现实机制或结构错位。
- 结构完整但没有真实洞见或现实落点。

## 3. 返工

先判断问题属于哪一类，再返工：

- **正文结构问题**：长表格、密集列表、标题层级混乱，回到 `delivery/final.md`。
- **封面信息缺失**：缺标题或副标题，回到 `delivery/final.md`，用 `# 标题` 后接 `> 副标题` 修正。
- **PDF 样式问题**：字号、行距、边距、引用块视觉重量，回到 `scripts/build-mobile-pdf.py`。
- **内容不够劲**：回到结构抽取或课程规划。
- **表达阻力太大**：回到正文做减法打磨。
- **边界不干净**：请 `references/agents/50-boundary-sanitation.md` 复查。
- **读者认知体验失守**：回创作或前置内容，重写入口和张力的来处，让阅读动力回到材料、旧模型、现实机制或结构错位。
