# 80 成品交付

## 0. 交付位置

本页说明主理人怎样把已通过的 `workspace/manuscript.md` 整理成 `delivery/final.md`，再生成手机阅读友好的 PDF。交付能不能出站，由 `references/standards/60-delivery.md` 裁决。

交付不是把过程文件丢给顾客，而是把成品清楚地交到顾客手里。默认最终交付适合手机阅读的 PDF；顾客明确要求 Markdown 时，可以同时交付整理后的 `delivery/final.md`。内容形态仍然只有单篇长文或多讲课程，PDF 和 Markdown 只是交付格式。

阅读体验优先。技术验证只说明文件生成成功，不说明它已经好读。排版也关系顾客能不能读懂：它决定顾客是否愿意读下去、能不能快速找到层级、会不会在密集信息里疲劳。

默认 PDF 的审美方向是“安静的精心准备”：像一份认真交到顾客手里的正式成品，而不是从 Markdown 直接打印出来的文件。

## 1. 整理最终文稿

从 `workspace/manuscript.md` 整理出 `delivery/final.md`。整理时只做交付必要统一：标题层级、衔接、目录、重复、未回收伏笔、明显语病和交付结构。不要把每一讲磨成同一种声音，不要把结构抽取、课程规划、创作日记或项目日志拼进成品。

`delivery/final.md` 是稳定整理稿，不是最终 PDF 的对外文件名。生成 PDF 时，输出文件名使用成品标题，并按需要带上讲数或形态，例如 `delivery/因果科学入门课-16讲.pdf` 或 `delivery/创造力长文.pdf`。不要用后台名 `final.pdf` 作为最终交付名。

标题结构：

- `delivery/final.md` 只有一个一级标题，且是整本产品标题；不要写成 `# 课程正文`、`# 最终稿`、`# final` 这类后台名称。
- 一级标题下一行写 `> 副标题`，副标题是给顾客看的封面承诺，不进入正文目录。
- 前言、发刊词、导论和各讲正文用二级标题。
- 导论内部小节和讲内小标题用三级标题或正文加粗。
- 结课地图是后台任务名，不直接作为读者目录标题。

生成前做一次书籍结构体检。体检只看成品骨架，不审内容优劣：

- 只有一个一级标题，且一级标题是整本产品名称。
- 一级标题下一行有 `> 副标题`，且副标题像给读者看的封面承诺。
- PDF 目录只收二级标题，所以二级标题必须像书的骨架。
- 已经有“第 N 讲 / 第 N 章”结构时，讲内小标题不要用二级标题；否则它会混进目录。
- 看脚本打印的 `PDF 目录预览`。如果这份列表不像读者会翻阅的目录，先回 `delivery/final.md` 修层级。

## 2. 生成 PDF

PDF 只走一个脚本：

```bash
python scripts/build-mobile-pdf.py delivery/final.md "delivery/标题名称-16讲.pdf"
```

脚本负责 Markdown 到 PDF、封面、目录、标题层级预检、CJK 字体栈、手机阅读版心、基础排版、目录预览和可复制正文。

脚本的视觉基准：

- 接近手机竖屏比例的页面尺寸。
- 暖白纸色、柔和正文色和克制强调色。
- 手机阅读友好的版心、字号、行距和段落间距。
- 封面保持干净；目录和正文底部显示页码，方便读者定位。
- CJK 字体栈。
- 保留正文里中英文交界处的作者空格。
- 列表、引用、表格、代码块做基础排版，但不抢正文重量。
- 生成前做书籍结构体检，发现缺少封面副标题、标题层级和目录边界问题时给出具体行号。

依赖缺失时，脚本会创建隔离运行时并重新运行。不要临时创建第二套 PDF 脚本；如果标准脚本不满足需求，优先修 `scripts/build-mobile-pdf.py`，让系统保持一个出口。

标准脚本核心依赖 Python 包 `markdown`，PDF 渲染优先依赖 `weasyprint`，兼容 fallback 依赖 `reportlab`。它不依赖 `markdown_it`。如果报错出现 `ModuleNotFoundError: No module named 'markdown_it'`，说明运行的不是当前标准脚本，或脚本被旧版本覆盖；先恢复 / 修正 `scripts/build-mobile-pdf.py`，不要为这次交付另写临时脚本。

## 3. PDF 验收

技术验收：

- PDF 文件使用成品标题命名，存在且非空。
- `pdfinfo "delivery/标题名称-16讲.pdf"` 可读取页数和尺寸。
- `pdftotext "delivery/标题名称-16讲.pdf" -` 可抽取正文。
- 抽取文本没有明显乱码、本地路径、`file://`、日期页眉页脚、调试信息或过程标记。
- PDF 标题是整本产品标题，不是发刊词、导论或前言。
- 封面有副标题；`delivery/final.md` 的一级标题下一行是 `> 副标题`。
- 目录里前置内容和正文章节层级正确。

阅读验收：

- 目录可跳转。
- 标题层级清楚。
- 字号、行距、页边距舒服，不像挤压后的打印稿。
- 封面安静、有留白，目录像目录，不像普通正文。
- 页面整体安静，不用大面积色块、重边框或过多高亮抢注意力。
- 段落不挤，列表换行对齐。
- 长表格没有横向挤压到不可读。
- 引用、代码块、表格是辅助信息，不喧宾夺主。
- 正文可以复制。
- 手机视角抽查阅读，不只看桌面预览；至少看封面、目录、普通正文页、列表页、表格或引用页。

`pdftotext` 只用于验证文本可抽取、中文无乱码、无 file URL / 本地路径残留。不要用它判断首行缩进、列表对齐、间距等视觉问题；视觉问题用 PDF 阅读器或截图看渲染结果。

## 4. 排版返工信号

- 看起来像 Word / Markdown 直接导出的文档。
- 正文一屏里黑压压一片，没有呼吸。
- 标题装饰太重，抢走正文注意力。
- 每个页面都像套了同一块色板，缺少细腻层级。
- 引用块、表格或代码块比正文更显眼。
- 长表格在手机上不可读；先回到 `delivery/final.md`，把表格拆成分组列表或卡片式表达，这不是 CSS 能解决的。
- PDF 好看但不像本产品：颜色、装饰或字体气质盖过了内容。

## 5. 非默认交付

过程材料、结构抽取、课程规划和创作日记可以作为生产现场保留，但不默认当作最终交付。顾客明确要“给创作继续用的材料”时，仍要整理成可读、可接手、边界清楚的文件，而不是把过程文件原样打包。
